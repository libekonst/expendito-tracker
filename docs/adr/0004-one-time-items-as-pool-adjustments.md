# One-time items adjust the pool, not a specific month

One-time Expenses and One-time Incomes (e.g. a trip, severance, selling a car) are not pinned to a calendar month. They adjust the Effective Balance — the total funds available before the simulation starts — and the engine runs from that adjusted number. The alternative was month-tied one-time items, where a trip in October would cause a step-down in the October balance. We rejected this because, with flat recurring amounts and no Activity Windows, the timing of a lump sum has no effect on total runway length; it only shifts where on the chart the step appears. Removing the month field eliminates a decision the user doesn't need to make and keeps the data model simpler. If variable monthly cash flows are ever introduced (e.g. Activity Windows return), this decision should be revisited.
